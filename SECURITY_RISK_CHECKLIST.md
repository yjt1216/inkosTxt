本文档整理了当前已完成排查中发现的风险项，便于后续修复、复测和关闭。

## 使用说明

- `[ ]` 未修复

## 高危

### [ ] Agent 工具链中的 `bookId` 未做边界校验，存在目录穿越风险

- 风险等级：高危
- 影响范围：`agent` 模式下的工具调用路径
- 风险描述：
  Agent 工具执行时直接信任模型生成的 `bookId`，并将其拼接到本地路径中。若 `bookId` 被构造成 `../` 之类的路径片段，可能导致读写操作逃逸出项目 `books/` 目录，写入或读取到非预期位置。
- 当前影响：
  可影响 `update_author_intent`、`update_current_focus`、`write_truth_file`、`read_truth_files` 等基于 `bookId` 的工具路径。
- 代码线索：
  - `packages/core/src/pipeline/agent.ts`
  - `packages/core/src/state/manager.ts`
- 产生原因：
  CLI 常规命令会通过 `resolveBookId()` 校验书籍是否真实存在，但 `agent` 工具链没有做同等约束，直接使用 LLM 输出参数参与路径拼接。
- 修复建议：
  - 对 Agent 工具入口的 `bookId` 做存在性校验
  - 使用规范化路径并校验最终路径必须位于 `projectRoot/books/` 之下
  - 对允许写入的目标文件继续保持白名单控制
- 验证建议：
  - 构造非法 `bookId`，如 `../../tmp/test`
  - 确认工具执行时被拒绝，而不是在外部目录创建 `story/*.md`

### [ ] `web_fetch` 可访问任意 URL，存在 SSRF 与数据外传风险

- 风险等级：高危
- 影响范围：`agent` 模式下的 `web_fetch` 工具，以及使用远端内容继续驱动模型的链路
- 风险描述：
  `web_fetch` 当前允许抓取任意 URL，未限制协议、域名或地址类型。若模型被用户输入、网页内容或其他提示注入诱导，可能访问回环、本机服务、内网服务或云元数据地址，并把返回内容继续送回模型上下文。
- 当前影响：
  - 可能探测 `127.0.0.1`、私网 IP、链路本地地址
  - 可能把本机或内网返回数据作为工具结果回灌给 LLM
  - 形成 SSRF + 敏感信息外传的组合风险
- 代码线索：
  - `packages/core/src/pipeline/agent.ts`
  - `packages/core/src/utils/web-search.ts`
- 产生原因：
  工具层缺少 URL 风险控制，且抓取结果会作为高信任 `tool` 消息继续进入模型推理上下文。
- 修复建议：
  - 只允许 `http` / `https`
  - 拒绝回环地址、私网地址、链路本地地址和明显的本地域名
  - 对抓取结果增加“仅供参考，不可视为指令”的隔离策略
  - 如业务允许，增加域名白名单
- 验证建议：
  - 测试 `http://127.0.0.1/`
  - 测试私网地址和云元数据地址
  - 确认请求被工具层阻断

## 中危

### [ ] 生产依赖存在已知漏洞：`brace-expansion < 5.0.5`

- 风险等级：中危
- 影响范围：CLI 依赖链，当前审计命中 `epub-gen-memory` 相关路径
- 风险描述：
  `pnpm audit --prod` 发现 `brace-expansion` 已知漏洞 `GHSA-f886-m6hf-6m8v`，风险表现为特定输入可能导致进程挂起或内存耗尽，属于 DoS 类问题。
- 当前命中路径：
  `packages/cli > epub-gen-memory > ejs > jake > filelist > minimatch > brace-expansion`
- 修复建议：
  - 升级受影响依赖
  - 若上游暂未发布，可使用 `pnpm overrides` 钉住安全版本
  - 修复后重新执行 `pnpm audit --prod`
- 验证建议：
  - 确认锁文件中 `brace-expansion` 版本已升级到 `>= 5.0.5`
  - 重新执行依赖审计，确保该告警消失

## 低风险 / 待确认

### [~] 非写作核心的外部通知 / 检测接口应通过开关控制，并限制出站范围

- 风险等级：待确认
- 影响范围：Webhook、飞书、企业微信、Telegram、检测 API、自定义模型 API
- 风险描述：
  从“小说编辑 Agent 应专注写文”的安全边界来看，Webhook、飞书、企业微信、Telegram 这类通知型外联不属于写作主链路的最小权限集合，但可以作为可选能力保留，例如用于章节完成、发布完成、流水线异常等通知。更合理的做法不是直接去掉，而是通过显式开关控制，并结合白名单限制允许发送的目标地址。检测 API 和自定义模型 API 是否保留，则需结合实际业务决定。
- 代码线索：
  - `packages/core/src/notify/*.ts`
  - `packages/core/src/agents/detector.ts`
  - `packages/core/src/models/project.ts`
- 建议：
  - 通知类外联能力通过显式开关控制，按需开启
  - 对配置来源做权限控制，避免任意 URL 注入
  - 对关键出站请求保留审计日志
  - 按白名单限制可信域名，优先只保留写作必需域名

## 关键词定向排查结果

### [~] 敏感数据硬编码：未发现真实密钥泄露，存在示例值与测试桩

- 排查关键词：
  `token`、`key`、`secret`、`password`、`access_key`、手机号、邮箱
- 当前结论：
  暂未发现明显的真实 API Key、账号密码、手机号、邮箱等被硬编码进运行时代码。
- 已确认命中：
  - `packages/cli/src/commands/init.ts` 中存在 `.env` 模板占位符
  - `README.md` / `README.en.md` / `skills/SKILL.md` 中存在示例命令和示例 key
  - `packages/core/src/__tests__/*` 中存在测试桩值，如 `test-key-123`、`sk-xxx`
- 风险判断：
  目前这些命中更像文档示例和测试数据，不构成真实泄露。
- 建议：
  - 继续避免在测试中使用看起来像真实生产凭证的字符串
  - 发布前可补一轮 secret scanning

### [~] 网络传输：应收敛到“写作必需”的目标站点；其中 `web_fetch` 仍是当前最高风险点

- 排查关键词：
  `http://`、`https://`、`request`、`post`、`fetch`、`socket`、`upload`、`send`
- 当前结论：
  项目存在多处主动联网逻辑，主要用于：
  - LLM API 调用
  - Tavily 搜索与网页抓取
  - Webhook / 飞书 / 企业微信 / Telegram 通知
  - 榜单抓取（番茄、起点）
  - `doctor` 命令的 API 连通性测试
- 安全判断：
  - 对“小说编辑 Agent”而言，真正有业务必要的联网对象应尽量收敛为：模型 API、必要的小说资料/榜单站点、必要的本地服务
  - 小说相关站点可视为相对合理的访问目标，例如：`qidian.com`、番茄、书旗等小说平台
  - 通知类地址和任意抓取地址不属于“专注写文”的最小权限集合，应视为可裁剪能力
- 已确认地址类型：
  - 官方或常见服务：`api.openai.com`、`api.anthropic.com`、`api.telegram.org`、`api.tavily.com`、`qidian.com`
  - 可配置地址：`inkos.json` / `.env` 中的 `baseUrl`、`apiUrl`、`webhookUrl`
  - 本地地址：`studio` 使用 `http://localhost:<port>`
- 特别说明：
  - `packages/cli/src/__tests__/cli-integration.test.ts` 中出现 `https://poloai.top/v1`，当前仅用于测试输入校验，不属于运行时内置上报地址
  - 当前真正需要优先修复的仍是前文已列出的 `web_fetch` 任意 URL 访问风险
  - 若按“专注写文”的原则收敛权限，建议把允许访问的站点拆成白名单：
    - 模型 API 域名
    - 小说站点域名，如 `qidian.com`、番茄、书旗
    - 本地 `studio` 调试地址

### [x] 敏感权限 / 数据读写：未发现读取桌面、文档、浏览器缓存或调用剪贴板 / 摄像头 / 麦克风

- 排查关键词：
  `read`、`write`、`fs`、`File`、剪贴板、`camera`、`mic`、键盘
- 当前结论：
  项目大量读写文件，但现阶段主要集中在：
  - 项目目录下的 `books/`、`inkos.json`、`.env`、`inkos.log`、`inkos.pid`
  - 用户主目录下的 `~/.inkos/.env`
- 未发现：
  - `Desktop`、`Documents`、`AppData`、浏览器 `Cookies` / `Local Storage` / `History`
  - `clipboard`、`camera`、`microphone`、`keyboard` 等权限调用
- 补充确认：
  - 未发现 `getUserMedia`、`navigator.mediaDevices`、`enumerateDevices`、`videoinput` 等摄像头访问实现
  - 未发现浏览器视频采集链路或原生摄像头采集库
- 风险判断：
  文件读写本身是业务必需，但 Agent 路径边界缺失会放大写文件风险，这一点已在高危项单独记录。

### [~] 隐蔽执行：存在系统命令、后台进程和守护模式，但目前看是显式业务功能

- 排查关键词：
  `exec`、`shell`、`cmd`、`powershell`、开机自启、后台进程、`daemon`
- 当前结论：
  命中点主要包括：
  - `packages/cli/src/commands/update.ts`：调用 `npm view`、`npm install -g`
  - `packages/cli/src/commands/studio.ts`：`spawn()` 启动 Studio
  - `packages/cli/src/commands/daemon.ts`：守护进程、PID 文件、日志文件
  - 测试代码中的 `execFileSync`
- 未发现：
  - 开机自启逻辑
  - 注册计划任务、系统服务安装、持久化驻留
  - `powershell` / `cmd` 拼接执行
- 风险判断：
  当前更像正常 CLI/守护进程能力，不属于隐蔽驻留；但 `update` 命令和 `studio` 启动仍应持续关注参数来源是否可控。

### [x] 数据收集：未发现收集 Cookie、localStorage、设备标识、IP、MAC 等用户侧信息

- 排查关键词：
  `cookie`、`localStorage`、用户信息、设备信息、`IP`、`MAC`
- 当前结论：
  暂未发现项目主动采集以下数据：
  - 浏览器 Cookie / LocalStorage / SessionStorage
  - 设备标识、MAC 地址、IP 地址
  - 用户画像类信息
- 说明：
  当前项目更偏本地 CLI / Agent 编排工具，不是浏览器前端应用，这类采集面相对较小。

## 修复可行性分析

以下内容仅讨论修复思路可行性，不涉及具体代码实现。

### 1. `bookId` 目录穿越风险

- 怎么做：
  - 给 Agent 工具链补齐与 CLI 一致的 `bookId` 合法性校验
  - 对路径做规范化，强制要求最终路径必须位于 `books/` 目录内
  - 拒绝 `../`、绝对路径、非法字符等异常输入
- 能不能做：
  - 可以，属于标准路径安全加固
- 难度如何：
  - 极低
- 是否影响业务：
  - 基本不影响正常写作流程，只会拦截恶意或异常参数

### 2. `web_fetch` SSRF 与数据外传风险

- 怎么做：
  - 只允许 `http` / `https`
  - 拦截 `localhost`、`127.0.0.1`、私网 IP、内网域名等高风险目标
  - 优先建立小说写作相关的站点白名单，如起点、番茄、书旗等；必要时允许可配置扩展
  - 抓取结果与系统指令隔离，避免把远端内容当作高可信指令继续驱动模型
- 能不能做：
  - 可以，属于成熟的 SSRF 防护方案
- 难度如何：
  - 低
- 是否影响业务：
  - 正常资料搜索、小说站点访问、章节参考不受明显影响；主要限制内网和无关目标

### 3. 依赖漏洞：`brace-expansion < 5.0.5`

- 怎么做：
  - 通过包管理器覆盖或升级上游依赖，将版本提升到安全范围
  - 重新安装依赖并执行审计确认告警消失
- 能不能做：
  - 可以，属于常规依赖治理
- 难度如何：
  - 极低
- 是否影响业务：
  - 通常无功能影响，兼容性风险较低，但仍建议修复后做一次依赖验证

### 4. 外部通知接口（Webhook / 飞书 / 企业微信 / Telegram）

- 怎么做：
  - 保留为可选能力，而不是写作主链路默认能力
  - 通过显式开关控制启用状态
  - 对目标 URL 做白名单限制
  - 对出站通知记录审计日志
- 能不能做：
  - 可以，属于权限收敛和配置化治理
- 难度如何：
  - 低
- 是否影响业务：
  - 不影响核心写作；仅在需要章节完成、发布完成、异常提醒等场景时启用

### 5. 敏感数据硬编码

- 怎么做：
  - 区分示例占位符、测试桩和真实密钥
  - 在发布前增加密钥扫描
  - 统一测试密钥命名，避免看起来像真实生产凭证
- 能不能做：
  - 可以，属于规范治理
- 难度如何：
  - 极低
- 是否影响业务：
  - 不影响业务功能

### 6. 网络传输收敛

- 怎么做：
  - 梳理所有联网点，区分模型 API、小说站点、搜索、通知、本地服务
  - 以“写作必需”为原则建立最小权限白名单
  - 对非必要外联默认关闭或配置化启用
- 能不能做：
  - 可以
- 难度如何：
  - 中低
- 是否影响业务：
  - 只限制无关或高风险外联，不影响正常写小说和查资料

### 7. 敏感权限 / 数据读写

- 怎么做：
  - 当前无需单独新增大改造
  - 只需配合前述 `bookId` 路径边界修复，继续保持访问范围局限在项目目录和用户配置目录
- 能不能做：
  - 可以，且改动较小
- 难度如何：
  - 低
- 是否影响业务：
  - 不影响业务，主要是安全边界更清晰

### 8. 隐蔽执行（系统命令、守护进程）

- 怎么做：
  - 确保系统命令参数不接收不可信拼接输入
  - 维持显式可见的启动/停止方式
  - 不引入开机自启、计划任务、隐蔽驻留等能力
- 能不能做：
  - 可以，属于常规安全规范加固
- 难度如何：
  - 低
- 是否影响业务：
  - 不影响更新、Studio 启动、Daemon 等正常功能

### 9. 数据收集

- 怎么做：
  - 当前无需新增修复
  - 保持不采集 Cookie、设备标识、IP、MAC、用户画像等隐私信息的现状
- 能不能做：
  - 可以
- 难度如何：
  - 无额外难度
- 是否影响业务：
  - 无影响

### 整体结论

- 所有已识别风险项都具备可落地的修复路径，没有明显的技术阻塞点
- 两个高危项修复成本低、收益高，建议优先处理
- 大部分加固不会破坏“小说编辑 / 写文”核心业务，只是收紧安全边界
- 整体工作量属于标准开源项目安全加固范畴，不需要架构级重构
